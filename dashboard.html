Arena Scheduling Comparison Vendor vs. PowerSchool
PowerSchool
Credit-Type Priority Model
Each credit type has its own defined priority chain. When a student earns a credit, it fills the first bucket in that credit type's chain that still has room. Overflow moves automatically to the next bucket in the chain.
Example — DS credit type:
DS → Computer Tech → CTE → Elective
The credit always tries Computer Tech first, regardless of what order requirements are listed.
vs
Arena Scheduling
Bucket Sort-Order Model
Requirements are processed in sort order (1 → 99). For each requirement, any credit the student holds that matches its accepted credit types is consumed there — in the order the requirements appear, not the order the credit type defines.
Example — DS credit type:
DS appears in CTE (sort 11) and Computer Tech (sort 14)
DS always fills CTE first because sort 11 comes before sort 14.
Concrete Example: Same Student, Different Results
A student completes two courses — one Digital Studies course (credit type DS) worth 1.0 credit, and one CTE course (credit type CTE) worth 1.0 credit. Two requirements still need credits: CTE (1.0 cr needed) and Computer Tech (0.5 cr needed).

Structural Impossibility
No Sort Order Can Fix This
Some of the conflicts above could be resolved by reordering requirements — but the problem below is mathematically irreconcilable. Two credit types both appear in the CTE and Science AA buckets, and PowerSchool routes them in opposite priority orders through those two buckets. A single sort order can only place one bucket before the other — it cannot satisfy both at once.

Sort "CTE first" fixes Student B but breaks Student A. Sort "Sci AA first" fixes Student A but breaks Student B. No sort order works for both.
Root Cause
PowerSchool lets each credit type define its own independent priority chain. Two credit types can point at the same pair of buckets and prefer them in opposite order — and PowerSchool handles that correctly because it evaluates each credit type's chain independently. Arena Scheduling enforces a single global sort order shared by all credit types. The moment any two credit types require opposite relative ordering for the same two buckets, no sort order configuration can satisfy both — the constraint is impossible to resolve within the model.
